# Prompts · 通用提示词模板

> 复制粘贴用。每个模板都是为某个固定场景设计的，**别混着用**。
> 模板里的 `{XXX}` 是你要填的，其他原样发。

---

## 模板 0 · 开 session（最常用，每次都用）

```
请先按顺序读：

1. CLAUDE.md（协作规则 + 末尾「当前进度」）
2. PROJECT_SPEC.md（产品 + 架构 + 当前阶段的 command/event/错误大类）
3. SESSION_PLAYBOOK.md 末尾的「中断信号」一节

读完用一段话回答我：

- 这个 session 我们做哪一个切片（切片号 + 一句话目标）
- 这个切片的验收条件是什么（按 SPEC 写的来，不要自己发挥）
- 你打算改哪些文件、为什么、对应 SPEC 哪一节
- 需不需要引入新库或新抽象

回答完停下，**等我说"开始"再写代码**。
如果上面任何一条 SPEC 没写清楚，直接告诉我哪里不清楚，我来补 SPEC，不要自己猜。
```

## 模板 1 · 出方案（模板 0 通过后，正式动工前）

```
按你刚才说的方案动手。规则：

- 一次只改一个文件 / 一个小功能，跑通后再开下一个
- 看不懂的代码不准写，要写就先解释 why
- 不要加 SPEC 没写的：错误处理、抽象、新文件、新依赖。要加先停下来问我
- 热路径里禁止 unwrap() / expect()
- 注释只写 why 不写 what

每写完一小段，**先解释一句"我刚做了什么、它对应 SPEC 哪一节"**，再继续。
我看着不对会随时打断。
```

## 模板 2 · 写系统层（快捷键 / VAD / 注入 / 流式）

```
这一段是系统层代码（{快捷键 / VAD / 剪贴板注入 / 流式 ASR}）。

按 CLAUDE.md 第 2 条规则：先读再写。

1. 去看 Handy 仓库（cjpais/Handy）对应实现：
   - 涉及到的文件是 {src-tauri/src/managers/XXX.rs} 之类
   - 把它的关键代码片段（30 行以内）贴给我看
   - 说一下它的思路、有什么坑、为什么这么做

2. 然后告诉我：我们的实现要不要照搬，还是有差异（比如要支持流式、要换平台）

3. 等我说"按 Handy 的来"或者"我们这样改" 再动手。

不许凭空生成系统层代码。
```

## 模板 3 · 我看不懂这段（中场用）

```
停一下，这段代码我没看懂。

按下面的格式给我讲：

1. 这段在做什么（一句话）
2. 为什么这么做，不这么做会怎样
3. 关键的 3-5 行是哪几行，分别什么意思
4. 如果未来要扩展 {XXX 场景}，要改这里的哪些地方

讲完我消化一下，看要不要让你重写得更简单。
不要直接重写，先讲。
```

## 模板 4 · 卡住了 / 螺旋下坠

```
停。我感觉我们在螺旋下坠。

按下面做，不要再写代码：

1. 用 git status 和 git diff 告诉我现在工作区有多脏
2. 我们最后一次能跑通的 commit 是哪个（git log 最近 5 条）
3. 这一轮我们尝试改了几次 {具体哪个问题} 都没成功
4. 你的诊断：根因是什么？是切片拆太大、SPEC 没写清楚、还是你方向走错了？

诊断完停下来等我决定回滚还是继续。
不许再写一行修复代码。
```

## 模板 5 · 收尾切片

```
我刚演示通过了这个切片。按下面收尾：

1. 列一下这次改动涉及的所有文件 + 每个文件改了什么（一句话）
2. 起草一个 Conventional Commits 的 commit message（中文，说 why 不说 what，标题 50 字以内）
3. 告诉我 CLAUDE.md 末尾「当前进度」该怎么更新（勾掉哪个切片，下次做哪个）

不要直接 git commit，等我确认 message 再执行。
```

---

## 模板 6 · weekly spec drift 检查（每周一次）

```
做一次 spec drift 检查，不要改代码。

1. 把 PROJECT_SPEC.md 的 §一.4「Non-Goals」逐条读一遍，对照 src/ 和 src-tauri/src/，看有没有偷偷长出 Non-Goals 里的东西
2. 把 PROJECT_SPEC.md 的 §三 / §四（当前阶段 command + event）逐条读一遍，看代码里实际的 command/event 命名和 SPEC 是否一致
3. 把 CLAUDE.md 的「当前进度」对照代码，看哪个 P0.X 是真做完了，哪个其实只做了一半但被勾掉了

输出一个 drift 报告：

- 已 drift 的项（代码和 SPEC 不一致）
- 漏勾 / 错勾的进度项
- 建议：是改 SPEC 还是改代码

不要自己改任何文件，等我决定。
```

## 模板 7 · 想改架构 / 想加新东西（先停下来用这个）

```
我在想 {要不要加 X / 要不要换 Y / 要不要新增一个 manager}。

按下面回答，**不要立刻给方案**：

1. 这个想法解决的是什么问题？这个问题现在有多痛、必须现在解决吗？
2. 不加这个，最简单的替代方案是什么？
3. 加了这个，要动 PROJECT_SPEC.md 的哪几节？会不会和现有架构铁律冲突（§六）？
4. 它属于 P0–P4 哪一阶段？提前做的代价是什么？
5. 在 typeless / Handy / Voxt 里，这个东西他们怎么做的？

回答完停下，等我消化。我大概率会决定「不加 / 推到 P3 / 先记 ADR 不动手」之一。
```

## 模板 8 · 让 AI 教我（非动工 session）

```
我想理解 {VAD 怎么工作 / Tauri command-event 怎么串 / macOS Accessibility 权限模型 / WebSocket 流式 ASR 协议 / ...}，今天不动代码。

按下面讲：

1. 从最朴素的版本讲起，假设我完全不知道
2. 关键概念 / 关键名词，用一句话定义
3. 在我们项目里，这个东西会落到哪个 manager / 哪个文件
4. 学的时候最容易理解错的点是什么

讲完出 3 个检查我有没有真的懂的问题，我回答你判分。
不要给完整代码，给伪代码或片段就行。
```

---

## 使用规律

- **每个 session 一定走模板 0 开场**。这是防止 AI 跳过 SPEC 直接写代码的闸。
- **模板 1 是模板 0 的延续**，一次会话里连着用。
- **模板 2 只在写系统层时用**，写普通业务代码不需要。
- **模板 3 / 4 是急救包**，出现就用。用早不用晚。
- **模板 5 是每个 session 的强制结尾**。少了它进度就乱。
- **模板 6 一周一次**，固定时间（比如周日早上）。
- **模板 7 / 8 是非动工 session**，不写代码，但和动工同样重要。

---

## 一句话总结

**模板 0 + 1 + 5 是地板，不许跳；其他按场景用。**

跳了地板模板，你又会回到"AI 替你做决策、生成你看不懂的代码"的老路。
